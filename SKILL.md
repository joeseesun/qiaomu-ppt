# Qiaomu PPT

Create presentation work that can actually be used: first route the job, then shape the story, then lock the visual system, then produce checked artifacts. Default to an editable PPTX-oriented path when the user says PPT / PowerPoint / 课件 / 二次编辑, and use HTML-only routes only when the user explicitly wants a web deck, a fast visual preview, or an interactive presentation.

## Runtime Dependencies

- Full editable PPTX delivery and PPTX preview verification require LibreOffice (`soffice`). On fresh installs, run `python3 <skill>/scripts/bootstrap.py --check`; if LibreOffice is missing, the agent should directly run `python3 <skill>/scripts/bootstrap.py --install-system` and rerun the check.
- `--install-system` installs required external tools from `data/dependency_manifest.json` with Homebrew on macOS or `apt`/`apt-get` on Ubuntu/Debian. For LibreOffice this means `brew install --cask libreoffice` on macOS or `sudo apt-get install libreoffice` on Ubuntu/Debian. Use `--install-system --include-optional-system` only when the project also needs optional tools such as Poppler, ImageMagick, Node.js, or archive extractors.
- LibreOffice official manual download page: https://www.libreoffice.org/download/ . Use it only when package-manager installation is unavailable, blocked by permissions, or explicitly declined by the user.
- If LibreOffice cannot be installed in the current environment, do not claim PPTX-to-PDF preview evidence or full PPTX QA. Continue only with explicitly degraded SVG/HTML artifacts and record the PPTX verification gap as `missing evidence`.
- Keynote export is optional and macOS-only. `scripts/export_bundle.py` always writes `keynote_capability.json` and includes a `keynote_capability` block in `export_manifest.json`, even when Keynote was not requested, so the project records whether the current machine can attempt Keynote automation. If modern `save as Keynote` automation is unavailable, blocked, or times out, `scripts/export_bundle.py` should try the Keynote 09 `.key` export fallback before marking the target failed. Do not claim Keynote compatibility from LibreOffice evidence. A successful `.key` must be a real artifact at least as fresh as the current PPTX source and must record whether it came from modern Keynote save or Keynote 09 fallback. For a reusable proof, run `scripts/keynote_smoke.py <project> --pptx <deck.pptx>` so Keynote export and `check_project.py` evidence are captured in `reports/keynote_smoke.json/md`. When Keynote still fails, run or report `scripts/keynote_probe.py --with-control` so the failure is split into launch, open/import, save, fallback export, cleanup, and baseline-control stages.

## Router Rules

- Start by writing a short route card before generating slides:
  - `route`: `brand_release`, `high_school_courseware`, `business_report`, `talk_deck`, `pptx_beautify`, `html_deck`, or `html_preview`.
  - `final_delivery`: `editable_pptx`, `semantic_html_deck`, `pptx_plus_semantic_html`, `preview_then_pptx`, or `planning_only`.
  - `confidence`: high / medium / low.
  - `assumptions`: only the assumptions that affect output.
- If the user says PPT, PowerPoint, PPTX, 课件, 可编辑, 二次修改, 老师要改, or deliverable is unclear, default to `editable_pptx`.
- Normal editable PPTX delivery must not be satisfied by inserting one full-slide PNG/JPG/PDF screenshot per page. Whole-slide raster PPTX is allowed only for explicitly labelled parity preview, social-image export, or user-approved non-editable visual draft, and the filename/report must say `image-backed` or `non-editable`.
- Do not create a visually separate low-fidelity "editable remake" after a good HTML/PNG preview. Use one shared `slide_plan.json` + `spec_lock.json` + `layout_execution_contract`: high-fidelity preview and PPTX export must come from the same layout/style contract. For polished editable PPTX, use the SVG-first/DrawingML route or a deliberate hybrid route where atmosphere/background imagery is raster, while titles, body text, numbers, charts, labels, cards, image frames, and diagrams remain editable foreground objects.
- If the user says 网页 PPT, HTML PPT, HTML 版本, 横向翻页, WebGL, interactive, or browser presentation, route to `html_deck` and deliver `semantic_html_deck` unless they explicitly ask only for a quick style preview.
- If the user says 先看风格, 快速预览, or visual direction preview, route to `html_preview` or `preview_then_pptx`; label it as preview, not final HTML delivery.
- Brand launch decks prioritize story tension, brand signal, first-slide impact, sparse copy, media decisions, and a polished visual thesis.
- High school courseware prioritizes learning objective, prerequisite check, concept ladder, examples, exercises, board-work rhythm, answer reveal plan, and teacher notes. Do not use a sparse keynote style for dense teaching unless the user asks for a public talk.
- If the user provides URLs, arXiv/Hugging Face paper links, WeChat public-account links, files, folders, ZIP source packets, PDFs, EPUB books, Office documents, Feishu docs, images, or mixed inputs, route through the source intake step first. Save clean Markdown, images, source metadata, and extraction gaps before writing slide claims.
- If the user provides only a broad topic, such as "制作一个 PPT 介绍蒲松龄", route through the topic research gate first. Use web research by default unless the user forbids it, save sources and source cards, then discuss 2-3 content angles with the user before writing the final design proposal. Do not generate slides from topic memory alone.
- For normal user-facing PPT creation, do not jump directly from source ingestion to generation. First present a design proposal and wait for user approval unless the user explicitly asks to skip approval or asks for an immediate draft.
- Do not call `ppt-master`, `baoyu-design`, `frontend-slides`, `guizang-ppt-skill`, `humanize-ppt`, or a standalone `pptx` skill as runtime dependencies. Their local clones/installed skills are research evidence only.
- Do not copy upstream templates, long wording, exact style packs, or copyrighted slide designs into generated artifacts. MIT-licensed infrastructure code may be vendored into `qiaomu-ppt` with NOTICE attribution, but generated projects should use Qiaomu-owned route contracts, visual systems, artifact contracts, scripts, and checks.
- Never write user-generated PPT projects, source captures, previews, or exports into the installed skill directory. The skill directory is for reusable instructions, scripts, templates, fonts, style presets, tiny docs assets, and test fixtures only.

## Design Proposal Gate

Before generating a normal PPT, provide a concise `PPT Design Proposal` and wait for approval. The proposal must include:

- `page_count`: target slide count and why.
- `source_research_summary`: for topic-only, mixed-source, or weak-source requests, summarize source coverage, source types, image availability, gaps, and the selected/user-confirmed content angle.
- `audience_state_change`: what the audience should understand, believe, or do differently.
- `story_arc`: 5-8 title-first beats, not generic agenda labels.
- `reference_model`: selected deck mode, visual style, image rendering/palette behavior, image/text layout vocabulary, and media placement logic. Keep mode, style, image rendering, and image placement as separate decisions.
- `style_candidates`: 3 visibly different candidates from the style library or visual systems, including palette, typography, layout mood, and why each fits or does not fit. Do not reuse the last successful style by inertia.
- `style_recommendation_reasoning`: show the score/reasons, source library, palette, typography, best-fit routes, and risk/avoid notes for each candidate. Do not merely say `recorded in style_recommendations.json`; the user must be able to choose or reject the style from the proposal text.
- `selected_direction`: the recommended style, with a short tradeoff note.
- `canvas_format_decision`: the target surface, selected from `ppt169`, `ppt43`, `xiaohongshu`, `moments`, `story`, `wechat`, `banner`, `a4`, or a custom canvas. Use `data/canvas_format_specs.json` and explain the current renderer boundary: final editable PPTX is proven for `ppt169` unless project-specific multi-canvas export evidence exists; other ratios are planning/HTML/social/poster contracts until verified.
- `layout_mix`: at least 5 distinct layout families for decks over 8 pages. Select concrete proof-structure IDs from `references/layout-pattern-library.md`, such as L01 hero claim, L08 comparison, L13 process flow, L18 mechanism loop, L20 chart-with-takeaway, L24 concept map, L31 objection/response, and L35 closing. For image-heavy, screenshot, quote, product, before/after, timeline, or data-context slides, also name concrete image/text IDs from `references/image-text-layout-patterns.md`, such as ITL03 full-bleed hero, ITL10 conclusion-first image, ITL13 before/after, ITL18 screenshot annotation, and ITL20 data plus context image.
- `image_text_layout_plan`: for media-rich decks, list the planned ITL pattern groups, image roles, crop/safe-area strategy, and text-over-image contrast strategy. Do not summarize this as generic "图文混排".
- `computed_image_layout_slots`: for user/source/web images that will sit side-by-side with text or in multi-image comparison grids, run `python3 <skill>/scripts/plan_image_layouts.py <image-or-dir> --format <canvas_format> --output <project>/image_layout_plan.json --markdown <project>/image_layout_plan.md` and carry the computed `image_area`, `text_area`, `preserve_aspect_ratio`, and `multi_image_plan` into `visual_contract.json` or `spec_lock.json`.
- `layout_clearance_contract`: for every media-rich slide, declare title/body/proof/image bounding boxes and a minimum clearance. Default minimum clearance is `48px` on `1242x1660` social canvases, `28px` on `1280x720` slides, or `0.22in` in PPTX units. If title/body/proof boxes overlap an image slot or fall below clearance, resize, move, wrap, or change the ITL pattern before export.
- `visual_components`: which slides need charts, conceptual diagrams, process maps, icon systems, generated images, or source screenshots. Abstract article decks should normally include 2-4 explanatory diagrams or visual models, not only text cards.
- `visual_asset_acquisition_plan`: classify visual assets as `ai`, `web`, `user`, `source`, `formula`, or `placeholder`; name expected counts, asset roles, rights/provenance policy, and which rows must be terminal before full rendering.
- `icon_watermark_decision`: whether semantic per-slide icon watermarks will be used. Default to `disabled` unless the chosen style has clear negative space and the watermark strengthens the reading path. If yes, describe the query strategy, preferred icon sets, opacity range, placement rules, and why this will not become decorative noise. If a preview review says the watermark is visually distracting, stop iterating on placement and set it to `disabled_for_this_style`.
- `image_generation_decision`: explicitly say whether built-in Codex image generation or a configured API backend will be used. If available, default to AI-generated atmosphere/background, chapter art, concept metaphor, scenario, object cutaway, texture, or moodboard candidates when they serve the story; do not generate fake evidence, fake screenshots, logos, charts, UI, or editable slide layout objects. If not used, explain why.
- `image_art_direction_plan`: for every AI-generated visual asset, define slide role, image role, `content_link`, `background_duty`, `semantic_anchor`, art direction, composition, camera/crop, material, lighting, safe area, foreground boundary, negative prompt, and replacement/import path. The generated bitmap must help the page's claim/proof/emotion become tangible; it is not a decorative wallpaper layer. Do this before asking Codex/gpt-image/another backend to generate images.
- `recommended_generation_stack`: when model choice is available, recommend Claude Opus 4.8 for reasoning/layout/code and gpt-image-2 or Codex built-in image generation for bitmap backgrounds/concept images; if unavailable, state the fallback model/image route. Record this in reports, not on visible slides.
- `background_plan`: number of backgrounds, role rhythm, image asset source, how each background supports page content, and how consecutive reuse will be avoided. When Codex image generation is available, prefer content-linked bitmap atmosphere/concept backgrounds over procedural linework; each selected background must record the slide claim/proof it serves, the concrete semantic anchor, and the foreground safe area.
- `font_and_keynote_plan`: font stack, whether fonts are installed/verified, and how Keynote/PowerPoint/LibreOffice compatibility will be tested.
- `deliverables`: PPTX, semantic HTML, parity preview, PDF/screenshots, speaker notes, and sidecar contracts.
- `model_contract`: whether the deck will write full `design_spec.md`, `spec_lock.md`, `slide_plan.json`, `visual_asset_manifest.json`, `assets/images/image_prompts.json`, and `visual_contract.json`. For URL decks, image-rich decks, chart/diagram decks, teaching decks, brand decks, and decks over 8 pages, the answer should be yes unless the user explicitly asks for a throwaway draft.

Only after the user approves or revises this proposal should the agent generate the full deck. For automation or batch mode, save the proposal as `<project>/design_proposal.md` and proceed only when the request explicitly says approval is not needed.

A requested page count, such as "10页" or "20页", is only a page-count constraint. It is not approval of visual style, narrative angle, canvas ratio, image-generation policy, or full-generation skip. Continue to ask for approval of the design proposal and, for decks over 7 slides, the four-slide preview unless the user explicitly says `不用确认`, `跳过预览`, `直接生成最终版`, or equivalent.

## Four-Slide Preview Gate

For any normal PPT expected to exceed 7 slides, do not generate the complete PPTX/HTML in one pass after proposal approval. First generate a 4-slide preview package and wait for user confirmation.

- Preview slides must cover four different roles, not four near-identical pages. Default choices: cover/opening, one dense proof/evidence page, one diagram/process page, and one breathing/quote/turning-point page.
- Prefer `python3 <skill>/scripts/prepare_deck_project.py ... --generate-preview --preview-decision pending` or `python3 <skill>/scripts/four_slide_preview.py <project>` to create the isolated four-slide SVG/PNG preview package. The script must write preview files under `<project>/previews/four_slide/` and temporary work under `<project>/_preview_work/`; preview SVGs are not formal final-deck SVG pages.
- When the user asks to match `ppt-master`, says the visual quality is poor, or the project is explicitly high-design/editorial, follow [references/ppt-master-grade-execution.md](references/ppt-master-grade-execution.md). In that mode, scripts may prepare contracts, assets, checks, previews, and export, but the four representative preview SVG pages must be authored or revised page-by-page from the current `design_spec.md`, `spec_lock.md`, `slide_plan.json`, and visual asset manifest. Do not satisfy this gate with a generic batch renderer or one repeated template family.
- The preview package must include editable PPTX or PPTX-oriented preview pages, semantic HTML preview when HTML is part of delivery, rendered thumbnails/screenshots, and `preview_gate.json`.
- `preview_gate.json` must record `mode: four_slide_preview`, selected slide numbers, output paths, visual decisions under test, known risks, and `user_decision`.
- Do not proceed to full generation until `user_decision` is `approved` or the user explicitly says to skip preview/full speed/batch mode. If skipped, record `skipped_by_user: true` and the exact user instruction.
- "生成一个 X 页 PPT" by itself is not an explicit preview skip. Treat it as a request to begin the production workflow, not permission to bypass style confirmation or the four-slide preview gate.
- If the four-slide preview reveals layout, connector, HTML readability, typography, background, or content-density issues, fix the skill/project rules before full generation. Do not patch only the preview pages.

## Compact Workflow

1. Inspect the source material and goal.
   - Read user files, pasted notes, links, old decks, images, brand material, syllabus, textbook sections, and constraints.
   - If this is a fresh install, run `python3 <skill>/scripts/bootstrap.py --check` before relying on URL/PDF/PPTX conversion tools. If required LibreOffice is missing, run `python3 <skill>/scripts/bootstrap.py --install-system`, then rerun `--check` before claiming PPTX preview/export evidence.
   - Choose or create a project output folder before ingestion. Default to the caller's current workspace under `outputs/<date>-<slug>/` when a workspace exists; otherwise use `~/Documents/Qiaomu PPT/<date>-<slug>/` for durable work. Use `~/Downloads/<slug>/` only for quick disposable exports. If the user explicitly names Desktop, Downloads, Documents, or an absolute path, honor that path.
   - If the environment has a GUI file picker or the user is not working inside a clear project/workspace, ask once where to save: current workspace, Documents, Downloads, or Desktop. If the user does not answer and the task should continue, choose Documents for durable decks and Downloads for temporary previews.
   - If URLs, arXiv/Hugging Face paper links, WeChat public-account links, local files, folders, ZIP source packets, PDFs, EPUBs, Office documents, Markdown/TXT files, or images are present, run `python3 <skill>/scripts/source_to_markdown.py <inputs...> --output-dir <project>/sources` before planning; add `--download-images` only when the source is known to be image-rich and bandwidth/time are acceptable. This writes `source_manifest.json`, extracted Markdown, extracted/copied visual candidates, and first-pass `source_notes.md` / `source_cards.json` by default. Local and remote PDFs should render the first few pages into `sources/images/pdf-.../page-*.jpg` with Poppler when available so source-page visuals can enter the asset queue; if rendering is unavailable, record `pdf_page_images_not_rendered` instead of silently continuing with text only. For a single simple non-paper/non-WeChat URL, `url_to_markdown.py` remains an acceptable lightweight path, but run `source_cards.py` afterwards when the deck will cite the source.
   - For ordinary topic/file/link requests where the user expects a complete PPT workflow, prefer the one-command wrapper `python3 <skill>/scripts/create_deck.py --topic "<topic>" <inputs...> --project <project> --slides <n>` for project preparation. Use `--produce` only after the design proposal is approved or the user explicitly requests a one-pass run; otherwise the wrapper stops after preparation and writes `deck_create_manifest.json` / `deck_create_report.md` with the next step. Under the hood it calls `prepare_deck_project.py`, then optionally calls `produce_deck.py` with the selected quality profile and formats.
   - Default complete-deck quality is `final`, not `professional` or `draft`: complete PPT requests should use `python3 <skill>/scripts/produce_deck.py <project> --quality-profile final --generate-images --enforce-quality-benchmark --fail-on-critical-repairs --benchmark-min-score 85` after the proposal/preview gates. The agent may downgrade only when the user explicitly asks for a quick rough preview, offline smoke test, or low-cost draft; in that case use `--quality-profile draft --no-generate-images` and label the result as draft. If a real image backend is unavailable, prepare `image_art_direction.json`, `assets/images/image_generation_queue.json`, and `assets/images/built_in_image_generation_guide.md`, then stop at `missing real image generation evidence` or explicitly label the output as a draft. A `ppt_master_grade` or final deck with no benchmark, no repair plan, no real/generated or source-mapped image density, or weak upstream audits must not be called complete.
   - Manual SVG/PPTX generators are allowed only for experimental previews or bespoke repairs. For normal final delivery, the manual path must still produce `reports/content_outline_audit.json`, `reports/element_plan_audit.json`, `reports/style_fit_audit.json`, `reports/visual_rhythm_report.json`, `reports/deck_quality_benchmark.json`, and `reports/deck_repair_plan.json`, and the benchmark must meet the declared quality target before reporting success.
   - For lower-level project preparation or troubleshooting, run `python3 <skill>/scripts/prepare_deck_project.py --topic "<topic>" <inputs...> --project <project> --slides <n>` directly. Add `--generate-preview` after approval or for regression smoke tests when a four-slide visual gate is needed. Add `--materialize-assets` when a fast local visual preview should create procedural PNG fallbacks for pending AI assets before preview rendering; record that these are preview fallbacks and replace with configured image-generation outputs for final-quality delivery. With inputs, it runs source intake, source cards, content contract, slide-plan seed, style/layout recommendation, visual contract, visual asset planning, selected-source-visual resolution, design proposal, and preview-gate setup. With only a topic, it now attempts fast automated topic research through `scripts/topic_research.py`, ingests selected web sources into `sources/source_manifest.json`, and proceeds to a source-backed outline when enough cards exist. It then downloads only the source images that `plan_visual_assets.py` selected for actual slide use via `scripts/resolve_source_visuals.py`, bounded by `--source-visual-limit` and `--source-visual-timeout`; use `--skip-source-visual-resolve` for fully offline or ultra-fast dry runs. If research fails or is skipped with `--skip-auto-research`, it writes a research brief and stops at `needs_research` instead of fabricating slide claims.
   - For arXiv or Hugging Face paper links, `source_to_markdown.py` automatically routes to `paper_to_markdown.py`: resolve the arXiv ID, prefer arXiv e-print TeX/source, extract figure/table evidence, emit `paper_manifest.json`, and use PDF fallback only with `missing_evidence` recorded.
   - For WeChat public-account links, treat generic URL extraction as a baseline only. If text is weak or images are missing, use a specialized open-source extractor from [references/wechat-source-intake.md](references/wechat-source-intake.md), request an export/copy, or record the missing evidence before slide planning.
   - For Feishu/Lark docs, use exported Markdown/DOCX/PDF/ZIP when provided, or an authenticated connector/API route when available. If access is unavailable, record `feishu_content_not_fetched` and ask for export/permission; do not pretend the doc was read.
   - For PDFs, EPUBs, papers, article pages, WeChat articles, Office docs, and image-rich pages, keep `source_manifest.json`, extracted Markdown, downloaded/copied images, rendered PDF page images, paper/WeChat sidecars, source cards, and missing-evidence notes as inputs for the slide plan.
   - For long books, documents, podcasts, or mixed source packets, consider the NotebookLM route from `qiaomu-anything-to-notebooklm` as an optional analysis accelerator: upload sources, ask deep-analysis questions, then save answers into `source_notes.md`. NotebookLM-generated slide decks are reference artifacts, not the final Qiaomu PPT structure.
   - If the user supplied only a topic, follow [references/topic-research-method.md](references/topic-research-method.md): write a research brief, search across biography/context/works/impact/images as appropriate, save `sources/source_manifest.json`, `sources/source_notes.md`, and `sources/source_cards.json`, then discuss 2-3 content angles with the user before writing the full design proposal. If web access is unavailable or the user forbids research, record `research_status: skipped_by_user` or `missing evidence`; do not fabricate authority from memory.
   - For broad-topic automation, run `python3 <skill>/scripts/topic_research.py "<topic>" --output-dir <project>/sources --depth fast` first; use `--depth balanced` or `--depth deep` only when the user explicitly wants broader research and accepts the extra time. Brave Search is used when `BRAVE_SEARCH_API_KEY` is configured; otherwise the script falls back to Wikipedia/Wikidata and records provider gaps.
   - Topic research must identify image/visual asset availability early. If source images are weak, decide whether to request user assets, use rights-clear public images, use generated concept imagery, or choose a text/diagram-led style.
   - For high school courseware, identify subject, grade, textbook/module, teaching duration, and whether it is new lesson, review, exam explanation, or open class.
2. Create the route card and deck contract.
   - Output route, audience, desired audience state change, final format, page count range, density mode, and verification plan.
   - Write an audience-purpose card: audience, purpose, desired action, current state, desired state, stakes, density mode.
   - For topic-only decks, pause after source notes and before design proposal to confirm the content angle, audience/depth, and image policy with the user. If the user explicitly asks to skip discussion, write the assumptions and continue.

## Route-Specific Rules

### Brand Release

- Strong default structure: hook, product/category tension, user pain, breakthrough, proof, product story, ecosystem/price/availability, closing line.
- Use fewer words and more deliberate visual hierarchy. One slide should have one dominant sentence or one dominant proof object.
- Preserve brand truth: use real product names, screenshots, logos, colors, and release facts only when verified or provided.
- Speaker notes should help the presenter sell the turn, not read slide copy aloud.
- Default final delivery remains editable PPTX unless the user asked for a web presentation.

### High School Courseware

- Start from teaching objective and student state, not visual style.
- Prefer a classroom sequence: lead-in, objective, prerequisite review, concept explanation, example, guided practice, independent practice, summary, homework/extension.
- Keep text readable from the back of a classroom; split dense content instead of shrinking it.
- Use diagrams, tables, and progressive reveal plans where they reduce cognitive load.
- Put teacher prompts, misconceptions, answer explanations, and board-work cues into speaker notes.
- Treat subject correctness as a hard gate. Pause if textbook edition, exam standard, formula, historical fact, or curriculum constraint is unclear and cannot be verified locally.

## Gate Ladder

Run these gates in order. Do not claim success before the relevant gates pass.

1. Route gate: route and final delivery match the user’s real need.
2. Source gate: every required claim, data point, image, brand asset, or textbook fact is either sourced, user-provided, or labeled as a draft assumption.
3. Topic research gate: broad-topic decks have research notes, source cards, image/material coverage, and a confirmed or explicitly assumed content angle; paper and WeChat source decks have extraction sidecars and gaps recorded before outline writing.
4. Content outline audit gate: `reports/content_outline_audit.json` scores source-backed claims, concrete anchors, claim-title quality, story contract completeness, source depth, and anti-generic copy. Professional/final runs fail early if this is weak.
5. Element plan audit gate: `reports/element_plan_audit.json` scores proof-object coverage, component-plan coverage, layout/media intent, element diversity, visual-asset queue readiness, and source-visual priority. Professional/final runs fail early if the deck is still text-only or decorative.
6. Style fit audit gate: `reports/style_fit_audit.json` checks whether the selected style matches the content domain, has real palette/typography/media/chart policy, covers every slide with a layout program, and respects source-first media policy.
7. Narrative gate: every slide has a clear audience/learning state change.
8. Copy gate: every normal slide has a claim title, a visible proof focus, and concise supporting copy.
9. Concreteness gate: every normal slide has at least one concrete anchor from the source or user brief: number, named example, mechanism step, boundary, counterargument, or consequence. A slide that reads like a generic slogan fails even if it is visually polished.
7. Layout principle gate: each slide declares a layout family and reading path; hierarchy, contrast, alignment, proximity, repetition, and whitespace support the claim. Media-rich slides also declare an ITL image/text pattern, image role, crop policy, safe text area, and contrast policy.
8. Visual asset acquisition gate: every visual asset row has a declared acquisition route, role, status, prompt/source, usage, rights/provenance notes, and file evidence where the status requires a file. Generated images serve atmosphere, concept, scenario, object, texture, or moodboard roles only; evidence images come from sources. AI background/concept rows must also declare `content_link`, `background_duty`, and `semantic_anchor`, tying the image to the page claim/proof or audience state change.
9. Visual gate: palette, typography, layout density, and media treatment are consistent; no generic purple-gradient default, no style soup.
10. Typography hierarchy gate: primary title remains visually dominant; if a support line is larger than the title, rewrite the title/subtitle relationship. CJK multi-line titles use readable leading; formal HTML title selectors below `line-height: 1.06` fail, and normal page titles should sit around `1.14-1.30`.
11. Background uniqueness gate: decks longer than 8 slides use role variants, do not put the exact same background asset on consecutive slides, and prefer unique background assets per slide up to 12 slides. The thumbnail grid must show clear visible differences, not only different filenames. Background differences must be content-led, not random decorative lines or unrelated abstract patterns.
12. Layout chrome gate: no dashboard-like card soup, no nested parent panels, no filler bottom cards, no repeated numbered badges without sequence meaning, and no more than 2 consecutive slides using the same evidence composition.
13. Shape component gate: card/callout/node text stays inside the shape with padding; connectors are simple and do not contain symbols or nested arrow glyphs.
14. HTML delivery gate: formal HTML decks are semantic web presentations, not whole-slide screenshots. They must preserve the same content contract as PPTX, use a fixed 16:9 stage with verified scaled centering, and pass per-slide browser screenshot readability. PPTX parity screenshot HTML is allowed only as `html_parity_preview` for QA or explicit pixel-match preview requests.
15. Provenance and language gate: source/model/toolchain evidence exists in sidecar artifacts, but internal QA metadata and production jargon are not printed on the slide canvas by default.
16. Readability gate: no text overflow, no overlap, no unreadable small type, no buried CJK punctuation issues obvious in preview.
17. Production gate: SVG/HTML/PPTX files exist in the declared output paths; export commands succeeded or are explicitly unavailable. When multi-format output is requested, `export_manifest.json` lists PPTX, PDF, formal HTML, parity HTML, and Keynote statuses separately.
18. Speaker gate: speaker notes or teacher notes exist when the route requires them.
19. Verification gate: preview/open/export checks ran; missing apps, credentials, fonts, or PPTX exporters are marked as `missing evidence`.
20. Keynote compatibility gate: if the user works on macOS or asks for Keynote, do not claim PPTX compatibility from LibreOffice alone. Install/verify declared fonts, test macOS Quick Look, try Keynote UI or automation when possible, and if Keynote fails, produce a diagnostic report plus a conservative fallback.

See [references/quality-gates.md](references/quality-gates.md).

- `reports/content_outline_audit.json` / `.md`, `reports/element_plan_audit.json` / `.md`, and `reports/style_fit_audit.json` / `.md`: upstream production gates that verify the deck has a sourced argument spine, proof-driven element plan, and content-fit style direction before rendering.
- `content_contract.json` or equivalent for decks longer than 8 slides: audience-purpose card, structure framework, title policy, evidence policy, and per-slide claim titles.
- `slide_plan.json` or equivalent structured plan. For topic-researched decks, mainline slides include `source_card_ids` and `source_anchor`.
- `page_content_guide.md` and `page_content_guide.json`: deck-level per-page content guide that combines each page's title, intent, visible points, source evidence, visual plan, visual assets, speaker notes, and QA summary in one easy-to-read place.
- `page_content/`: one editable Markdown file per page, named `01-<page-title>.md`, for manual review, rewrite, teaching notes, and later regeneration.
- `style_recommendations.json` when style is unspecified or the user asks for automatic style selection.
- `layout_recommendations.json` when the deck is image-rich and image/text composition is unspecified; selected `ITLxx` patterns must flow into `slide_plan.json`, `spec_lock.json`, and `visual_contract.json`.
- `image_layout_plan.json` and optionally `image_layout_plan.md` when real source/user/web images need side-by-side or multi-image placement; this records target canvas format, source image ratios, computed image/text slots, fit policy, and multi-image grid/collage handoff.
- `visual_asset_rows.json`: deterministic slide-plan-to-asset acquisition queue produced by `scripts/plan_visual_assets.py` before normalization into `visual_asset_manifest.json`.
- `visual_asset_manifest.json`: every generated/web/user/source/formula/placeholder visual asset row, acquisition route, status, prompt/source, file path, role, usage, rights/provenance, and QA notes.
- `assets/images/generation_batch/`: optional staged real-image-generation package created by `scripts/stage_image_generation.py`, containing prompt files, negative prompts, metadata, request JSONL, generated-output folder, and import mapping.
- `assets/images/image_generation_run.json` when `scripts/run_image_generation.py` planned or executed a configured image-generation run.
- `assets/images/image_prompts.json` and `assets/images/image_prompts.md` when AI-generated visuals are planned; these are the image-generation work queue and human-readable sidecar.
- `assets/images/image_sources.json` or source-specific sidecars when web/user/source assets are used, recording URL/file origin, license/rights notes, and extraction gaps.
- `style_brief.md`: visual thesis, tokens, layout rhythm, media policy.
- `style_direction.json/md`: executable art-direction contract derived from style/layout recommendations, including asset-density targets, media/chart policy, non-negotiables, and slide-level layout program.
- `reports/source_adequacy.json/md`: source-material sufficiency report covering source count, text volume, source cards, image candidates, source-like visual assets, and style evidence-page targets; professional/final production must not pass when core source evidence is too thin.
- `layout_variation` or equivalent: per-slide layout/background roles, image-text layout plan, and image slot contract.
- chart and diagram source specs under `<project>/charts/` or `<project>/diagrams/` when charts/diagrams are generated.
- `shape_component_policy` inside `visual_contract.json`: safe area, text-fit rules, connector grammar, operator placement, card density, separator usage, and preview rejection policy for out-of-bounds components.
- `assets/icons/` and icon usage records when semantic SVG icons are used.
- `icon_watermark_plan.json` when large per-slide semantic watermark icons are used, including query, provider, selected icon, opacity, placement, rationale, and rejection notes.
- `visible_provenance_policy` or equivalent: whether sources/citations appear on slides, in notes, in a final references page, or only in QA.
- deck source files, such as SVG pages or HTML preview.
- `pptx_generation_manifest.json` when the built-in PPTX exporter is used: generator path, source contracts, output path, slide count, and editable policy.
- `svg_generation_manifest.json` when SVG-first pages are generated: generator path, method, slide count, source directory, and recommended next commands.
- `svg_preview_manifest.json` when SVG previews are rendered: per-slide PNG paths and thumbnail grid.
- `exports/<slug>.pptx.trace.json` when SVG-to-PPTX conversion trace is enabled.
- `pptx_preview_manifest.json` when PPTX visual preview is rendered: PDF path, slide image paths, thumbnail grid, and preview tools.
- `pptx_text_check.json` when PPTX visible text is inspected.
- exported `.pptx` when the route is editable PPTX and an exporter is available; otherwise checked SVG/HTML artifacts plus `missing evidence`.
- exported `.pdf` when portable viewing is requested or produced from PPTX preview/export evidence; otherwise a PDF `missing evidence` entry in `export_manifest.json`.
- exported `.key` only when Keynote automation succeeds on macOS and the package is at least as fresh as the current PPTX; otherwise Keynote is recorded as `missing evidence` or `failed`, not implied by PPTX/LibreOffice success.
- exported `.html` semantic web presentation for `html_deck` and `pptx_plus_semantic_html` routes, unless the user explicitly forbids HTML output or the route is planning-only.
- `html_delivery_manifest.json` for formal HTML decks: mode, slide count, source contracts, output paths, component strategy, accessibility/searchability notes, and proof that the visible layer is not whole-slide screenshots.
- `html_parity_manifest.json` only when a preview is generated from rendered PPTX/PDF slide images for exact visual QA. Its outputs should use `html-parity/` or `.parity.html` names and must not be reported as the formal HTML deck.
- `export_manifest.json` when `scripts/export_bundle.py` is used: requested formats, output paths, status per format, degradation reason, and the rule that formal HTML and parity HTML are distinct deliverables.
- `reports/*.keynote-probe.json` when Keynote export fails and deeper diagnosis is needed; it records environment, launch, open/import, save, freshness, cleanup, and a minimal native PPTX control probe when `--with-control` is used.
- speaker notes / teacher notes when expected.

- `reports/deck_quality_benchmark.json` / `.md`: quality benchmark against the ppt-master learning catalog, including score, category evidence, warnings, and baseline statistics.

- `reports/deck_repair_apply_report.json` / `.md`: deterministic contract repair preflight report when `deck_repair_apply.py` or `produce_deck.py --auto-apply-repairs` is used.

## Pause Conditions

Pause and ask the user before continuing if:

- required proprietary brand assets, copyrighted source files, paid APIs, credentials, or account actions are needed.
- the task requires factual claims about a product, law, medical/scientific fact, exam standard, or current event that cannot be verified from supplied sources.
- a user asks for pixel-perfect PPTX equality with a complex HTML/WebGL deck; explain that editable PPTX and web rendering have different tradeoffs.
- an editable PPTX exporter is unavailable but the user requires a real `.pptx`.
- three focused QA/fix rounds still leave narrative, readability, or export failures.
